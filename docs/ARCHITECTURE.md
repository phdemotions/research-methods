> Design documentation for the gold-standard research methods skill suite.
> Read this before building or modifying any skill.

## Core Philosophy

### The Dual Problem

Most researchers face two failure modes:
1. **The SPSS refugee** — has data, knows the statistics, but produces code that can't be reproduced, reviewed, or published.
2. **The AI-assisted disaster** — uses AI to generate code that looks competent but is generic, untraceable, and would embarrass anyone who published it on OSF.

This skill suite solves both: it guides researchers toward code that is both methodologically rigorous AND publication-ready, while being specific to their research question, not boilerplate.

### The Three Non-Negotiables

1. **Raw data is sacred.** Never modified, only read. Every transformation is logged and reversible. Raw data lives in `data/raw/` and is read-only.
2. **One command reproduces everything.** The pipeline (`targets` in R, `Snakemake`/`Makefile` in Python) runs every step from raw data to final output. `tar_make()` or `snakemake` — that's it.
3. **Every subjective decision is documented.** The `docs/decisions/` log captures exclusion criteria, outlier handling, variable transformations, model specification choices, and deviations from pre-registration — with rationale.

### What makes this not "generic AI"

Every skill is **project-aware**. When a skill runs, it:
- Reads the pre-registration (if it exists) to know what was planned
- Reads the codebook to understand variable types, scales, and constructs
- Reads the decision log to understand prior choices
- Names variables using the actual constructs from the study
- Writes comments that reference the research question, not textbook language
- Flags deviations from the pre-registration and documents them as such

The difference between "run a regression" and "test H2: the moderated mediation of brand authenticity on purchase intention through perceived value, conditional on consumer expertise (PROCESS Model 14)."

## Skill Design Pattern

Every skill follows the same structure, mirroring the audit suite pattern:

```
skill-name/
├── SKILL.md              # Thin entry point with frontmatter
└── references/           # Loaded only when skill runs
    ├── principles.md     # Core principles for this stage
    ├── criteria.md       # Specific checks/requirements
    └── templates/        # Output templates (if applicable)
```

### SKILL.md Frontmatter

```yaml
---
name: skill-name
description: >
  Trigger-rich description used for auto-invocation.
  Includes: primary use case, trigger phrases, what stage of research this covers.
argument-hint: "<research question or project path>"
---
```

### Progressive Disclosure

- SKILL.md is loaded on every conversation startup (keep it thin)
- `references/*` is loaded only when the skill actually runs
- `_shared/*` provides cross-cutting standards loaded by reference

## The Self-Improvement Mechanism

`/research-zeitgeist` is the engine that keeps the suite current.

### How it works

1. **Reads today's date** from the environment context.
2. **Web-searches** for:
   - New CRAN/PyPI releases for all packages in FRAMEWORKS.md
   - Updated methodology guidelines (ASA statements, APA updates, PRISMA updates)
   - Methodology blog posts from recognized sources (Andrew Gelman, Frank Harrell, Julia Silge, Hadley Wickham, etc.)
   - Recent R/Python conference talks on research methods
   - Changes in journal requirements (JCR, JMR, JCP, Management Science, AMJ, ASQ)
3. **Compares** current skill recommendations against findings.
4. **Produces a dated report**: what's new, what's changed, what skills need updating.
5. **Optionally proposes updates** to FRAMEWORKS.md and skill reference files (requires approval).

### Monthly cadence

Run `/research-zeitgeist` at the start of each month. The skill timestamps every finding so you can track the evolution of best practices over time.

## Integration with Existing Opus Vita Skills

These skills are **standalone** — they live in their own repo and can be published independently on GitHub. They do NOT depend on the Opus Vita audit suite or session hooks.

However, if working within the `~/developer/` monorepo:
- The session-end hook and `/session-audit` still fire normally
- `/zeitgeist` (app-focused) and `/research-zeitgeist` (methods-focused) are complementary
- Research projects within the monorepo get both layers of quality assurance

## Data-Already-Collected Workflow

A primary use case: researchers who come to this suite with data already collected and need to make it gold-standard.

The workflow is:
1. `/research-init` — scaffold the project structure around existing data
2. `/data-validate` — run quality checks, generate codebook, identify issues
3. `/data-clean` — produce documented cleaning scripts with CONSORT-style flow
4. `/eda` — exploratory analysis to understand what you have
5. `/analyze` — confirmatory analysis matched to hypotheses
6. Continue through the pipeline...

The key insight: even with existing data, the documentation, validation, and transparency infrastructure adds enormous value. A well-documented, reproducible analysis of existing data is worth more than a sloppy analysis of perfectly collected data.

## Formatting Standards

### Default: APA 7th Edition

- Statistics: *F*(2, 147) = 4.32, *p* = .015, eta-squared = .06
- Effect sizes always reported alongside p-values
- Confidence intervals preferred over point estimates
- Exact p-values (not p < .05) unless p < .001
- Tables follow APA table format (gtsummary/modelsummary handle this)

### Journal-Specific Overrides

Skills accept a `journal` parameter that adjusts formatting:
- `journal: "jcr"` — Journal of Consumer Research
- `journal: "jmr"` — Journal of Marketing Research
- `journal: "jcp"` — Journal of Consumer Psychology
- `journal: "ms"` — Management Science
- `journal: "amj"` — Academy of Management Journal
- `journal: "asq"` — Administrative Science Quarterly

Default is APA 7th. Override when submitting to a specific journal.
